# SFT sweep — rejection-sampling teacher trajectories (2026-06-14)

**Goal:** SFT Qwen3.5-4B on 27B-teacher CodeAct trajectories (rejection-sampled,
strict ANLS==1.0, all 426 trajectories kept), sweep LoRA rank × LR, pick the best
config, full-val it vs the base-4B baseline.

> ⚠️ **Read first — the merge/serve bug (now fixed).** Every SFT eval run *before*
> the evening of 2026-06-14 silently evaluated the **base model**. `verl.model_merger
> merge` on a LoRA run writes base `model.safetensors` **plus a separate
> `lora_adapter/` subdir** — it does *not* fold the adapter in; serving that dir with
> plain `vllm serve` (no `--enable-lora`) loads base and drops the adapter. So the
> earlier "config ranking", "LR×rank×epoch ladder", and "lr4e4 cap-relief" numbers
> were all base-4B — "SFT ≈ base" was tautological. **Fix:** fold the adapter with
> PEFT `merge_and_unload()` and *assert the merged weights differ from base*
> (`outputs/_fold_lora.py`). **All numbers below are the CORRECTED, properly-folded
> evals** (merged ≠ base verified per config). Lesson captured in CLAUDE.md
> "Evaluation methodology".

## Setup

- **Data:** `data/sft/teacher_pool.parquet` — 426 teacher trajectories (anls==1.0),
  page mix incl. real multi-page docs; token lens median ~2.3K, max ~23K.
- **Training:** 4-GPU FSDP, flash-attn + remove-padding + dynamic-bsz@24576, GDN
  kernels (fla + causal-conv1d — the ~12–18× throughput fix), LoRA all-linear,
  8 epochs, constant LR, ckpt every 2 epochs (ep8 = global_step_208), effective
  batch = 16 samples. ~7s/step, ~26 min/config. (Training was never affected by the
  merge bug — only eval was.)

**Loss-curve diversification (why 6 configs, not 9).** Three configs were skipped
as near-duplicates by their *training loss curve* — a cheap proxy that avoids
spending VLM-bound eval budget on redundant runs:
- `sft-r64-lr1e4` (loss 0.044) ≈ `sft-r64` (0.040): **same rank, near-identical
  curve** — the LR difference washed out. Exact dedup.
- `sft-r16-lr1e4` (0.108) ≈ `sft-r32-lr5e5` (0.132): redundant point in the gentle
  band, off the clean rank-32 axis.
- `sft-r32-lr5e5` (0.132): single gentle-band representative kept as `lr1e5` instead
  (more distinct). *(Note: r32 & r16 at LR2e-4 have different losses 0.051 vs 0.065
  yet identical 23.1% eval → rank is eval-irrelevant here, confirming the dedup.)*

**Read:** `sft-r64` is the clear screen leader (30.8%, 0 cap), the only config
clearly above the ~19–22% base band. Everything else clusters at 23.1% ≈ base. But
30.8% = 4/13 questions at n=1 — within the ±14pp noise, so this names a **full-val
candidate**, it is not a verdict. The loss→eval relation is noisy, not a clean
inverted-U (lr4e4 has nearly the lowest loss but a low eval; r64 has *the* lowest
loss and the best eval) — confirming loss is a *screening* proxy only.

## Full-val (headline) — leader sft-r64, n=1

Full 80-question val set, **n=1** (VLM perception throughput is the hard ceiling;
n≥4 was infeasible in the time budget). Cap-relief (timeout 2400, conc 16), VLM
served **with prefix caching** (the survey→verify loop re-looks at the same pages,
so image-prefix KV reuse is a real speedup). The 13 rank13 results were cloned in as
a head-start (rank13 ⊂ full-val verified), so only the remaining 67 questions ran.

| model | full-val ANLS | n | notes |
|---|---|---|---|
| base-4B baseline | 0.2234 ± 0.0344 | 8 | established baseline-to-beat |
| sft-r64 ep8 (n=1) | 0.287 | 1 | submit-only 41.8% · iter_cap 26% — **favorable single draw, see below** |
| **sft-r64 ep8 (n=8, matched)** | **0.230** | 8 | pass@8 53.8% · SC@8 17.5% · submit-only 34.9% · iter_cap 30% |

**Result (matched n=8 — the authoritative number): sft-r64 = 23.0% avg@8 vs base-4B
22.34% (n=8) → +0.7pp, a TIE within the base's ±3.44 noise.** pass@8 53.8% ≈ base 55.0%
(tied); **SC@8 17.5% < base 26.25% (worse)** — SFT reduced answer self-consistency. So at
matched power **SFT (r64) does NOT beat base — it ties on mean/pass@8 and is worse on SC@8.**

> ⚠️ **The earlier n=1 "+6.4pp" was a measurement artifact — corrected here.** The n=1 number
> (28.7%) used only sample_idx=0, which happened to be a *cleaner draw* (26% iter_cap). The full
> 8-sample average regresses to base (30% iter_cap, submit-only 41.8%→34.9%). Comparing n=1-SFT
> to n=8-base was apples-to-oranges; the matched n=8 (23.0%) is the honest comparison and it says
> **tie**. This is consistent with the entire prior SFT investigation (SFT-on-teacher-trajectories
> ties base) — now confirmed with a correctly-folded checkpoint at matched power. The lever to
> move the metric is **on-policy RL**, not SFT-on-trajectories.

**Note — matched-n resolves the earlier caveat.** An earlier draft flagged that the n=1
SFT number was a lower bound vs the n=8 base (apples-to-oranges). That caveat is now moot:
the **n=8 matched run above is the apples-to-apples comparison**, and it says tie. Read it
as overall + submit-only + cap, not overall alone (iter_cap 30% is the headroom, not capability).

## Verdict (final)

- The prior "SFT ≈ base" rested on a measurement bug (base-vs-base). The bug is fixed
  (correct LoRA fold), but **the corrected, matched-power answer is STILL "SFT ≈ base":**
  at n=8, `sft-r64` = **23.0% avg@8 vs base 22.34%** (+0.7pp, tied within ±3.4pp),
  **pass@8 53.8% ≈ 55.0%** (tied), **SC@8 17.5% < 26.25%** (worse). So SFT-on-teacher-
  trajectories does not beat the untrained base at matched power — it ties on the mean and
  loses self-consistency. (The intermediate n=1 "+6.4pp" was a favorable-draw artifact,
  retracted above.)
- This *confirms* the entire prior SFT investigation with a correctly-folded checkpoint:
  multi-turn observation-shift defeats trajectory imitation; SFT is a safe warm-start at
  best, not a standalone win. **The lever is on-policy RL** (GRPO from base, launching next).
- Headroom is **iter_cap (30%)** — the agent exhausts its turn budget on hard multi-page
  docs before submitting (a turn-budget/perception-efficiency issue, addressable in RL via
  an efficiency signal).

## Notes / gotchas

- **LoRA fold:** `verl.model_merger merge` does NOT fold the adapter (writes base +
  `lora_adapter/`). Fold with `outputs/_fold_lora.py` (PEFT `merge_and_unload` + assert
  merged≠base) and serve the folded dir with plain `vllm serve`. Always weight-diff
  merged-vs-base before trusting any fine-tuned eval.
- SFT checkpoint layout is FLAT (`global_step_N/model_world_size_*.pt`, no `actor/`).
- eval.py `--model` is both the API id AND the tokenizer source — pass a real path.
- eval.py supports **sample-level resume** (`--resume` + `--rerun-terminations
  wall_cap,token_cap`) and a head-start clone (copy `tasks/<doc>/trajectories.jsonl`
  into a fresh run dir) — used here to reuse the rank13 rollouts.
- Eval/RL are VLM-perception-bound: ~24 concurrent rollouts saturate the 27B; more
  just starve each rollout (→ wall_cap). Prefix caching reduces per-look prefill.
